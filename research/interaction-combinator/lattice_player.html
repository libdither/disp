model + trace semantics
This page replays recorded engine state; it never simulates. Every lattice site is one 64-bit word — matter 20 + control 28 + χ 8 + σ 8 — updated in place by the six-neighbor center rule. Each sweep activates its cells in a fair deterministic random permutation (live-read: later activations see earlier writes from the same sweep), one activation per recorded tick. Scenarios come from the Cell64 test suite; regenerate the bundle with cargo run --release --bin dump-suite from evaluators/rust-ca-lattice/crate/.

AGENTS & PORTS

Port 0 is always the principal (▸): producers emit their value there, consumers consume there, and a fire needs the two principals facing across one edge. Aux ports carry everything else. On the canvas a white triangle marks the principal face, a hollow circle marks aux 1, and a diamond marks aux 2. When both auxiliaries use one face, their markers sit side-by-side over the same cable. Producers use four-sided diamonds; consumers use six-sided faceted diamonds.

REWRITE RULES

consumer principal ↔ principal producer ⇒ fresh agents (the dying pair's external wires resplice per the ROM's wiring permutation). Gold = firing in the frame you are on; ×n = firings in this trace (dimmed = never fires here); click a rule to jump to its next firing.

WHAT THIS SHOWS (AND DOES NOT)

Each scenario records one suite run activation by activation: frame 0 is a full keyframe, later frames are literal one-cell deltas plus their activation and fire events. Must-tier scenarios assert invariants (transport bounds, control clearance, matter preservation); example-tier scenarios record reductions such as the A·F rewrite — request the preferred lift, fall back when blocked, then place the fixed 16-cell result. At control-free checkpoints the lattice is projected to the shadow interaction net and checked port-for-port. The viewer only replays; the white activation marker makes each sweep's fair random order visible.
