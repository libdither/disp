 - It feels strange that disp has so far not required metavariables for anything. My understanding is that these are usually needed to make dependent type checking actually like readable instead of having annotations everywhere. Why hasn't disp needed it? Does this discrepancy potentially signal a major issue? Will disp need it eventually? If so, in what contexts?

   Resolution (investigated 2026-07-08): not a warning sign. Metavariables are an artifact of inference in an elaborator, not a requirement of dependent typing. Every mainstream core (Coq/Agda/Lean) is metavariable-free; only the elaborator on top uses them to fill in what the user omitted. Disp's elaborator is metavariable-free too, because it does no inference: it check-compiles fully-declared terms to trees (annotations erased at `src/elab/expr.ts:117`; holes parsed at `src/parse.ts:318` then rejected at `src/elab/expr.ts:91`) and defers all typing to the runtime kernel. There is no infer-mode that synthesizes a subterm's type bottom-up, so none of the infer-meets-check junctions that spawn metavariables exist. An un-annotated `let` is just an untyped tree.

   The three usual sources of metavariables get three different answers:
   1. Conversion / definitional equality: eliminated architecturally. Eager normalization + O(1) hash-cons `tree_eq` means we never compare a term-with-a-hole, only concrete normal forms or rigid neutrals. This is permanent.
   2. Implicit arguments: erased dependent intersections (the Cedille / Implicit-Calculus-of-Constructions lineage; `Intersection` at `lib/kernel/base.disp:56-59`, used by `param_apply` at `engine.disp:179-183`). The term does not abstract over the type argument, so it already inhabits every instance and instantiation is definitional. Nothing to solve, so no metavariable, and polymorphic calls pass no type argument.
   3. Dependent index refinement: no dependent pattern matching. Elimination goes through recursors (`rec_value`/`elim`) whose motive is applied by substitution, so there are no index equations to unify.

   Disp keeps the rigid half of the standard machinery (neutrals / eigenvariables, minted by `bind_hyp`) and omits the flexible half. Even the occurs-check is repurposed from assignment-guard to scope-escape (`engine.disp:87-104,143-149`).

   The cost is real and is the "annotations everywhere" the question senses: no ML-style principal-type inference, and generic code stays explicit. The pressure is deferred, not dissolved. It will resurface as (a) implicit inference for generic libraries (wanting `let y := map f xs` typed without writing the dependent instance), (b) typed holes for interactive work (the `_` stub already parses then errors), and (c) bidirectional checking to cut annotation burden.

   Likely disp-native answer: keep the kernel metavariable-free forever and add hole-filling above it as scored search rather than classical unification. The pieces already exist. Optimizer rewrite rules are schematic `metavars -> (lhs, rhs)` matched by `tree_eq`, not unification (`lib/tests/opt_q1_cert.test.disp:9`), and `std/demand.disp` already solves dependent constraints (`{x : Nat, y : Eq Nat x 1}` extracts exactly `{x := 1; y := refl}`) via marker instantiation + hash-consing + search. Unification is just the decidable, most-general, single-solution corner of the `GOALS.md` synthesis engine, which is a coherent reason it was never the priority.



 - A type system I'm told should have two properties, "soundness" (if you put in two syntactically equal things, you get equal outputs), completeness (if they have the same normal form, they are syntactically equal), and round-trip (if you eval a normal-form, it should get back itself). And using these, you can decide equality.
