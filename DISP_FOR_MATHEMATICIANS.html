Disp

A first-principles walkthrough for mathematicians — from λ-calculus to a self-hosting dependent type system on tree calculus, with a glance at the categorical and cubical horizons.

1.  Motivation

1.1  What is Disp?

Disp is a dependently typed programming language whose computational substrate is tree calculus rather than λ-calculus, and whose central design principle can be stated as a single equation:

T (v) = TT   ⟺   v : T

Read this aloud: "the type T applied to the value v reduces to TT if and only if v inhabits T."

In Disp, types are not a distinct stratum sitting above terms, related by a typing judgement Γ ⊢ v : T. Types are tree-calculus programs — predicates that, when run against a tree, return Boolean TT or FF. Typing is literal program execution.

Quote (README.md:5) — "Tree calculus is natively reflective — terms ARE data, so the type checker, the optimizer, and the programs it produces all inhabit the same universe. Disp's goal is a fully self-hosting dependent type system whose checker lives as an inhabitant of the object language, not just as a host-language function."

1.2  Why does this design exist?

The long-term aspiration, stated in GOALS.md, is a self-improving optimizer: given a formal specification of "good program," synthesize the actual program. This requires every component — the specification, the type-checker, the scoring function, the candidate programs — to live inside a single, simple, fully-reflective computational universe. Lean, Coq, and Agda have not bootstrapped their checkers into themselves because their checkers depend on host-language features (mutable hash tables, exceptions, pattern-match compilation, tactic DSLs) that have no object-language counterpart. Disp's development philosophy — quoted from DEVELOPMENT_PHILOSOPHY.md:11&12 below — refuses this drift from day one.

"The object language is the specification. Host implementations are optimizations. Every component that participates in deciding well-typedness — the type checker, the elaborator's semantics, conversion, the data structures those components read — must have a declared encoding as a tree-calculus program. The TypeScript implementation exists to make iteration fast. It is never the ground truth." — DEVELOPMENT_PHILOSOPHY.md:11–13

The reward for the discipline is that the type checker, eventually, can synthesize itself. Disp is built backwards from this goal.

2.  From λ-calculus to Trees

We begin where the working mathematician begins: untyped λ-calculus. The terms are

M, N  ::=  x  |  λx. M  |  M N

and the single computation rule is β-reduction, (λx. M) N →β M[x ≔ N]. The system is Turing complete and elegant, but it has a feature that pinches when you try to build a fully reflective language: λ-terms cannot inspect their own structure. Given a λ-term f, there is no β-reducible context in which one can ask "is f a function or an atom?" — everything is a function.

2.1  The SKI detour

Curry and Schönfinkel observed that one can dispense with bound variables entirely using three combinators:

CombinatorDefinitionBehaviour
Iλx. xI x = x
Kλx. λy. xK x y = x
Sλx. λy. λz. (x z)(y z)S x y z = (x z)(y z)

SKI is combinatorially complete: every λ-term has an equivalent SKI translation (the algorithm is called bracket abstraction). But SKI still has λ-calculus's opacity problem — you cannot ask "is f an S or a K?" as a reducible operation.

2.2  Tree calculus

Tree calculus, due to Barry Jay, is a small alternative inspired by SKI but with a crucial twist: every value is a tree built from three node shapes, and there is a primitive rule that pattern-matches on those shapes. The carrier is the free magma on a single nullary constructor — call it △ (a triangle, "leaf") — with binary operation written by juxtaposition. So every tree is one of:

△       (leaf, arity 0)
△ u      (stem, arity 1, holds one child)
△ u v     (fork, arity 2, holds two children)

In Disp's implementation (src/tree.ts:4–16) this is the inductive type:

type Tree =
  | { tag: "leaf"; id: number }
  | { tag: "stem"; child: Tree; id: number }
  | { tag: "fork"; left: Tree; right: Tree; id: number }

Application is not a syntactic constructor; it is an external operation apply : Tree × Tree → Tree. Every tree is in normal form on its own — reduction only happens through apply.

2.3  The reduction rules

The five rules of apply(f, x) partition by the shape of f (and, in the third case, by the shape of x):

Pattern of fResultName / interpretation
△△ xbuild a stem
△ u△ u xbuild a fork
△ △ bbK rule (constant)
△ (△ c) b(c x)(b x)S rule (composition)
△ (△ c d) btriage on x: leaf→c, stem(u)→d u, fork(u,v)→b u vF rule (triage)

Pause on the fifth rule. It is what distinguishes tree calculus from λ-calculus and from SKI. A program of shape △ (△ c d) b is a case analysis on the structure of its argument. Triage is the primitive reflection: every tree can be decomposed into its constituent shapes, by reduction alone.

Why this is a big deal. In λ-calculus, there is no β-reducible predicate "x is a variable" or "x is a λ-abstraction." In tree calculus, you can write is_leaf as a closed tree program, and reduce it against any value to get TT or FF. This is what is meant by structural completeness: every program can decide the structure of every other program by reduction alone.

2.4  Bracket abstraction: compiling λ to trees

Just as SKI admits bracket abstraction, so does tree calculus. The translation [x].M — remove the variable x from M, leaving a closed tree expression — is implemented in src/compile.ts:77–102:

function abstractName(name, body) {
  if (!containsFree(body, name)) return cap(K, body)         // [x].C  = K C
  switch (body.tag) {
    case "var": return I_CIR                                // [x].x  = I
    case "app":
      // η: [x].(f x) where x ∉ f   →   f
      if (body.x.tag === "var" && body.x.name === name && !containsFree(body.f, name))
        return body.f
      const af = abstractName(name, body.f), ax = abstractName(name, body.x)
      // K-composition:  S (K p) (K q)   →   K (p q)
      if (af.tag === "app" && af.f.tag === "K" && ax.tag === "app" && ax.f.tag === "K")
        return cap(K, cap(af.x, ax.x))
      return cap(cap(S, af), ax)
  }
}

The three combinators I, K, S are themselves specific trees:

I  =  △ (△ △) △     K  =  △ △     S  =  △ (△ (△ △)) △

So every λ-term in the surface syntax of Disp becomes — via this bracket abstraction — a closed tree. There are no free variables in the runtime; binders are compiled away.

2.5  Hash-consing and O(1) equality

One implementation detail is load-bearing for the whole design. Disp's tree runtime stores every distinct tree exactly once: when fork(l, r) is invoked twice with the same children (identified by their integer ids), the second call returns the same node as the first (src/tree.ts:26–48). This is hash-consingstructural sharing: identical subtrees share one heap cell. Consequence: structural equality is identity:

export function treeEqual(a, b) {
  return a.id === b.id   // O(1) — no traversal
}

Why is this load-bearing? Because conversion checking in a dependent type system is structural equality of normal forms. By making conversion O(1), the kernel keeps type-checking cheap. The whole "types are predicates that compare via reconstruction" trick — we will see this is the H-rule — only works because deterministic elaboration produces identical trees for identical types, and identical trees have identical ids.

3.  Types as Predicates

3.1  Propositions, predicates, and Curry–Howard

The mathematician already knows three closely related notions of "type":

  1. Set: T is a collection; v ∈ T is membership.
  2. Predicate: T is a function from a universe U to {true, false}; v ∈ T when T(v) = \text{true}.
  3. Proposition (Curry–Howard): T is a proposition; v : T is a proof.

Standard Martin-Löf type theory chooses (3). Set theory chooses (1). Disp chooses (2), with U the set of all trees and "true" the canonical tree TT:

T : Tree → Bool,   v : T  ⟺  T(v) reduces to TT

Aside: predicates vs. propositions, and why this is not just "set theory in disguise"

Choosing predicates over propositions looks, at first glance, like discarding proof-relevance. But Disp's predicates are run on tree-shaped values that include proof terms. The type Eq A x y, for instance, is a predicate over candidate proofs: it answers TT precisely when the candidate has the shape of a reflexivity certificate and the two endpoints are structurally equal. So the proof-relevance survives — it's just relocated from the typing judgement into the predicate's body. The advantage: there is no separate object called "the typing judgement"; there is only function application.

3.2  Booleans as functions: the Scott encoding

To make T(v) = TT a literal equation, Disp encodes Booleans not as opaque values but as their own eliminators. The encoding is Scott-style: a Boolean takes a motive and two branches, and dispatches to the appropriate branch (lib/prelude.disp:19–20):

TT := {m, ct, cf} -> ct
FF := {m, ct, cf} -> cf

After bracket abstraction these compile to specific trees: TT = K K = fork(△, stem(△)) and FF = K (K I). The combinator select is then trivial:

// select then else cond  =  cond t then else
select := {then_v, else_v, cond} -> cond t then_v else_v

If cond is TT, applying it discards the motive t and returns then_v. There is no if-keyword and no opcode; conditional dispatch is just Scott dispatch.

3.3  The central equation, in code

Suppose Nat is some tree (a predicate on trees). Then to check that 0 has type Nat, we simply apply:

Nat 0   ⇝   evaluates to TT or FF

There is nothing else — no typing context to maintain, no judgement to derive. The type checker is reduction. The library will define Nat shortly. First we need to confront the only real difficulty: what happens when v is not a closed value but a variable?

4.  The Seven Kernel Primitives

4.1  The hypothesis problem

Consider f := \{x\} \to x, the identity function, and ask: does f have type Pi Nat (\{n\} \to Nat)? Following our central equation, this becomes "evaluate (Pi Nat ({n} -> Nat)) f and check it returns TT." But to verify that f produces a Nat for every input, we must — somehow — consider all possible inputs.

The standard move in dependent type theory is to introduce a variable x : Nat in the typing context and check f\;x : Nat under that hypothesis. Disp does the same thing operationally: it mints a fresh tree that stands for "an unknown value of type Nat" and runs f on it. This minted tree is called a neutral or hypothesis, and it has a very specific shape:

Hyp Nat id  =  wait(kernel.hyp_reduce)(neutral_meta)

where neutral_meta packages the stored type (here, Nat) with a unique identifier and a "spine" recording how the hypothesis has been applied so far. Crucially, the head signature kernel.hyp_reduce marks this tree as a kernel-minted neutral — not user code — and the kernel uses this signature to dispatch the type-checking machinery.

Soundness threat. If a user could fabricate their own tree masquerading as a neutral — or could decompose a real neutral with triage to inspect its stored type — type checking would be unsound. Imagine: "give me a hypothesis h of type False; now apply the predicate False to h; it reports TT by H-rule, so I have proved False." The kernel must prevent this.

4.2  The parametric walker

The kernel solves the soundness threat by partitioning all apply calls into two modes:

  • Raw mode. The five reduction rules as written above. Used inside the kernel itself, where total visibility is required.
  • Parametric mode, also called the walker. The same five rules, but with two restrictions: (i) the F-rule (triage) on a hypothesis-rooted argument x fails; (ii) the stem-rule cannot fork into a shape whose head signature matches a kernel-minted neutral.

The walker is the discipline of operational parametricity: user code, running under the walker, cannot examine the structure of a hypothesis, and cannot forge new ones. From the user's perspective, a hypothesis behaves exactly like an opaque universally quantified value. The kernel's mode-switching is invisible to the user, but it is what makes f(h) — for a freshly minted h — type-correct iff f is correct on all inputs.

Quote (TYPE_THEORY.typ:147–149) — "Parametric mode encodes operational parametricity: hypotheses are opaque tokens for universal quantification, and user code cannot inspect their structure or synthesize new hypotheses."

4.3  The seven primitives

Everything else about the type system is library code. The kernel exposes exactly seven primitive handlers, packaged as a fixed-point record (lib/kernel/handlers.disp:340–348):

kernel : {hyp_reduce, guard, unguard, checked_apply,
          predicate_frame, eliminator_frame, bind_hyp} := recq {
  hyp_reduce       := q_hyp_reduce_fn;
  guard            := q_guard_fn;
  unguard          := q_unguard_fn;
  checked_apply    := q_checked_apply_fn;
  predicate_frame  := q_predicate_frame_fn;
  eliminator_frame := q_eliminator_frame_fn;
  bind_hyp         := q_bind_hyp_fn
}

Each handler does one thing.

① checked_apply — the dispatcher.

The gatekeeper of execution. Every application apply(f, x) in user code goes through checked_apply. It inspects the signature of f's head; if the signature matches one of the seven handlers, the handler runs in raw mode; otherwise the walker takes over. This is the privilege boundary: kernel handlers are raw-privileged, user code is parametric.

② hyp_reduce — what happens when you apply a hypothesis.

If h is a hypothesis of stored type T, what does h(arg) mean? The kernel consults T's metadata for a codomain function, which returns an Action: either Extend(B) — "the result is a new hypothesis of type B" — or Return(v) — "the result is the concrete value v." The new hypothesis's spine records the application history. We discuss Actions below in §4.4.

③ guard — the public boundary.

Every published type-former wraps its predicate in guard. When the predicate is invoked at the boundary — that is, when a user writes T v at top level — guard first scans the candidate v for any pre-existing kernel-minted neutrals (handlers.disp:100–108):

let q_guard_fn = {ks, raw, query} ->
  {core, v} ->
    match (q_scan_no_neutral raw v) {
      TT => match (tree_eq (core v) TT) { TT => TT; FF => FF }
      FF => FF
    }

If v contains a stray hypothesis, guard rejects with FF before evaluating the predicate. This closes the smuggling attack: hypotheses cannot leak across the public boundary into closed proofs.

④ unguard — the trusted peel.

Sometimes library code needs to unwrap a guarded type to access its inner predicate (e.g., to compose a new type former from existing ones). Doing this naively would re-trigger the walker, which forbids inspection. unguard is the handler that performs the peel in raw mode, recognized by signature and dispatched safely.

⑤ predicate_frame — the H-rule.

This is the engine of all type-checking. It wraps a user-supplied recognizer with a special case for hypothesis inputs: if the candidate v is a hypothesis, do not run the recognizer; instead, compare v's stored type against the type reconstructed from this predicate's metadata. This is the H-rule (hypothesis rule), and it is sound because hash-consing makes "reconstructed type" and "stored type" structurally compared via O(1) identity.

let q_predicate_frame_fn = {ks, raw, query} ->
  fix ({self, meta, v} -> {
    let check_fn = {ks, raw, query, self, meta, v} -> {
      let recognizer = pair_fst meta
      let params     = pair_fst (pair_snd meta)
      let result_r   = ks.checked_apply (recognizer params) v
      must_ok_or_ff result_r ({result} -> result)
    }
    select q_h_rule_fn check_fn (q_is_neutral raw v) ks raw query self meta v
  })

⑥ eliminator_frame — induction on hypotheses.

The dual of predicate_frame. Recognizers check membership; eliminators destruct. If the target of an eliminator is a hypothesis, we cannot pattern-match the case — we don't know which constructor it is. Instead, eliminator_frame mints a stuck neutral whose stored type is the motive applied to the target. Later, when the hypothesis is instantiated, the stuck neutral resumes reduction.

⑦ bind_hyp — binders that mint hypotheses.

The mechanism by which dependent types like Pi A B verify "for all x : A, the body holds." bind_hyp mints a fresh hypothesis of domain type A, runs the body with it, then performs an escape check: if the body's result contains the minted hypothesis via any non-neutral path, bind_hyp rejects (handlers.disp:188–214). The escape check is what prevents the bound variable from leaking into a closed proof.

The escape check, explained

The danger: suppose bind_hyp mints h with stored type False, the body returns h itself, and the caller now possesses h : False — a proof of contradiction. The escape check walks the result and asks: is the freshly minted h reachable via a non-neutral path? If so, the binder rejects. Hypotheses can occur inside other neutrals (that's how spines accumulate) but never as a directly extractable value.

4.4  Actions and spines

The Action protocol (lib/kernel/utils.disp:78–93) lets a type's codomain function communicate the result type back to the kernel without the kernel knowing the type's internal structure:

extend_tag := t              // the leaf △
return_tag := t t            // the stem △△

Extend := {v} -> t extend_tag v
Return := {v} -> t return_tag v

For example, the codomain function inside Pi (lib/types/pi.disp:34–40) is:

let pi_pf_codomain_fn = {ks, raw, neutral_meta, arg} -> {
  let stored_type = q_unguard_or_self ks (neutral_meta_type neutral_meta)
  let T_meta      = type_meta stored_type
  let params      = pair_fst (pair_snd T_meta)
  let B           = pair_snd params
  Extend (B arg)                          // codomain at this argument
}

When the kernel evaluates h(arg) for h : Pi\ Nat\ (\{n\}\to Nat), hyp_reduce calls Pi's codomain_fn, gets back Extend(Nat), and constructs a new hypothesis with stored type Nat and spine extended by arg. This is how spines grow without the kernel hardcoding a notion of "function application."

4.5  Three layers of soundness

The kernel's soundness theorem (informally stated): if T(v) = TT evaluates at the public boundary, then v inhabits T per T's predicate. It rests on three runtime mechanisms:

  1. Public-boundary scan. guard rejects candidates that contain pre-existing kernel-minted neutrals (the "smuggle a hypothesis past type checking" attack).
  2. Walker triage rejection. User code cannot perform triage on a hypothesis. This prevents extracting the stored type and is what makes parametricity operational.
  3. Stem-rule rejection. User code cannot fabricate a fork whose head looks like a kernel-minted hypothesis. This prevents forging neutrals.

One soundness carve-out exists: the polymorphic identity I reduces against any argument, including hypotheses. This is necessary because Pi\ Type\ (\{A\}\to Pi\ A\ (\{x\}\to A)) — the identity-function type — must be inhabited.

What the kernel deliberately does not guarantee: termination of user predicates, semantic correctness of user-supplied recognizers, detection of paradoxical type definitions. These responsibilities sit with the library author. Russell-style paradoxes such as R := {T} -> not (T T); R R are well-formed in Disp's syntax but diverge under evaluation, and the apply budget catches divergence as failure rather than synthesizing a proof of False.

5.  Library Types

Pi, Bool, Nat, Eq, Type, Ord, Sigma — none of these are kernel primitives. They are library definitions built from predicate_frame and eliminator_frame with type-specific recognizers and codomain functions. The kernel does not grow as new types are added.

5.1  Bool and Nat

From lib/types/bool.disp:

let bool_pf_recognizer = {_, v} -> or (tree_eq v TT) (tree_eq v FF)
Bool := guard (predicate_frame_form (t bool_pf_recognizer (t t t)))

That is the entire definition of Bool: "a tree is a Bool if it equals TT or FF." When applied to a hypothesis h, the H-rule inside predicate_frame bypasses the recognizer and structurally compares stored type against reconstructed type. Bool is library code; the kernel has never heard of it.

Natural numbers (lib/types/nat.disp:14–26) use a recursive predicate:

let nat_pf_recognizer = fix ({self_recog, params, v} -> {
  let nat_inner = predicate_frame_form (t self_recog (t t t))
  match (is_leaf v) {
    TT => TT                                   // zero is △
    FF => match (is_fork v) {
      TT => and (is_leaf (pair_fst v))
                (nat_inner (pair_snd v))      // succ n = fork(△, n)
      FF => FF
    }
  }
})
Nat := guard (predicate_frame_form (t nat_pf_recognizer (t t t)))

A Nat is either a leaf (zero) or a fork whose left is a leaf and whose right is a Nat. The recursive call goes through predicate_frame_form so that when the recursion encounters a sub-hypothesis, the H-rule fires.

5.2  Dependent functions

The Pi type-former is library code that uses bind_hyp to mint a hypothesis and run the body:

let pi_pf_recognizer = {params, v} -> {
  let A = pair_fst params
  let B = pair_snd params
  bind_hyp A ({hyp} -> {
    let result        = v hyp
    let expected_core = B hyp
    expected_core result
  })
}

Pi_pf := {A, B} -> guard (predicate_frame_form
  (t pi_pf_recognizer
     (t (t (unguard_checked A) ({x} -> unguard_checked (B x)))
        pi_pf_codomain_fn)))

Reading: "to check that v : Pi A B, mint a fresh hypothesis hyp : A, apply v to it, then check the result against the codomain B(hyp)." Cleanly stated, this is just the standard rule for dependent function types. But it is library code, not kernel code.

5.3  Propositional equality

refl := t        // the leaf △ is the canonical proof of reflexivity

let eq_pf_recognizer = {params, proof} ->
  and (tree_eq proof refl)
      (tree_eq (eq_meta_lhs params) (eq_meta_rhs params))

Eq := {A, x, y} ->
  guard (predicate_frame_form
    (t eq_pf_recognizer (t (make_eq_meta (unguard_checked A) x y) t)))

A proof of Eq A x y is the leaf, and only when x and y reduce to identical trees (O(1) comparison). The eliminator eq_J follows the standard Martin-Löf pattern, implemented via eliminator_frame.

Mathematician's eye. Disp's propositional equality is, in the present design, definitional: Eq A x y is inhabited iff x and y are convertible (= identical after normalization, given hash-consing). Cubical extensions, sketched below in §7.2, replace this with a path-based equality where univalence becomes a theorem.

5.4  The universe

There is a single universe Type; Disp does not stratify universes. A value inhabits Type iff it is structurally a published type-former — i.e., a guard-wrapped predicate_frame with well-formed metadata (lib/types/type.disp:71–96):

let type_pf_recognizer = {_, v} ->
  match (has_sig kernel_ref.guard v) {
    TT => match (has_sig kernel_ref.predicate_frame (type_meta v)) {
      TT => type_pf_metadata_well_formed (type_meta (type_meta v))
      FF => FF
    }
    FF => FF
  }

Without stratification, paradoxical definitions exist as syntactic objects but diverge: R := {T} -> not (T T); R R loops, and the runtime's apply-budget catches the loop as failure. Soundness is achieved by divergence-as-failure rather than by Tarski-style hierarchy.

6.  What does Disp look like to write?

The surface syntax is small. Here is a tour, drawn from SYNTAX.typ and lib/std/nat/arith.disp:

// Binders use braces; arrows are unicode-free
id : {A : Type} -> A -> A
   := {A} -> {x} -> x

// Pattern-matching on Bool
not := {b} -> match b {
  TT => FF
  FF => TT
}

// Addition by induction
add := {n, m} -> nat_rec ({_} -> Nat) m ({pred, ih} -> succ ih) n

// Compile-time test (a proof obligation discharged by reduction)
test (Arrow Nat Nat) ({x} -> x) = TT
test add (succ (succ t)) (succ t) = succ (succ (succ t))

The compiler (src/compile.ts) elaborates this through a small number of passes:

  1. Parse. Tokenize and parse to an AST.
  2. Elaborate. Resolve names, dispatch {x} -> e by mode — in term mode it compiles to a λ-abstraction, in type mode it wraps in a Pi.
  3. Bracket-abstract. Translate λs to combinator trees using the K-composition and η optimizations from §2.4.
  4. Emit. Produce a closed tree.

The elaborator never checks types; it produces tree programs, and the kernel checks by reduction. This is the metacircular discipline of DEVELOPMENT_PHILOSOPHY.md in action: elaboration is just translation, and type-checking is application.

Trace: checking the identity at Pi Nat (λ_. Nat)

Goal. evaluate (Pi Nat ({_} -> Nat)) ({x} -> x) and expect TT.
Step 1. Pi Nat (λ_.Nat) reduces to guard (predicate_frame_form …). The outer apply enters guard.
Step 2. guard scans the candidate {x} -> x for stray neutrals. None found. Passes through.
Step 3. Core predicate is predicate_frame wrapped around pi_pf_recognizer with params (Nat, λ_.Nat).
Step 4. Candidate is not a hypothesis, so dispatch to the recognizer.
Step 5. Recognizer calls bind_hyp Nat (λhyp. body). Kernel mints fresh h := Hyp Nat 0.
Step 6. Compute ({x} -> x) h. Walker reduces this under parametric mode. Result: h.
Step 7. Compute expected: (λ_.Nat) h = Nat. Check Nat h.
Step 8. Nat is predicate_frame-wrapped. Candidate h is a hypothesis. H-rule fires: compare reconstructed type (Nat) against stored type (Nat). Identical — TT.
Step 9. Escape check: does the result TT contain h? No. bind_hyp returns Ok(TT).
Result. TT. ∎

7.  Future Directions

7.1  Categorical foundations

CATEGORY_THEORY_FOUNDATIONS_PROPOSAL.typ argues that the existing seven-primitive design is already a faithful implementation of standard categorical machinery — the proposal is to make this explicit by typing the type-former metadata. A type's three metadata slots correspond to:

  • Recognizer  ⟷  a characteristic morphism χ_T : Tree → Ω into the subobject classifier (Ω = Bool).
  • Codomain_fn  ⟷  the evaluation morphism of a locally cartesian closed category, or the membership relation of a topos.
  • Comp_fn (proposed, cubical)  ⟷  the morphism action of a functor between ∞-groupoids.

The proposal refactors the opaque 3-tuple into a typed record TypeFormer with named fields, so that library authors must construct identity and composition laws as propositional proofs — functoriality is enforced by the kernel.

"The only genuinely Disp-specific decision in the entire categorical picture is 'everything is a tree' — the commitment to Tree as the universal object. Everything else slots into well-established categorical frameworks (topos theory, LCCC, ∞-category theory)." — CATEGORY_THEORY_FOUNDATIONS_PROPOSAL.typ:285–294

Because Disp's walker enforces parametricity, the relevant categorical model is a parametric topos — a PER model or reflexive-graph model where parametricity refines which characteristic morphisms count as valid types. This places Disp on familiar mathematical ground: standard topos theory applies with this refinement.

7.2  Cubical type theory and univalence

The two cubical proposals (CUBICAL_PROPOSAL.typ and the introductory CUBICAL_TYPE_THEORY_FOR_DUMMIES.typ) argue that the move from MLTT-style propositional equality to path-based equality — the source of computational univalence — can be implemented as library code over the existing seven primitives. No new kernel work is required.

The interval and paths

I is a library type whose values form the free De Morgan algebra on two constants I_zero and I_one with operations I_and, I_or, I_inv. It is not Boolean: in particular i I_or (I_inv i) ≠ I_one in general, preserving the continuous interpretation. Disp's walker already enforces the parametricity discipline that paths require — one cannot triage on an abstract i : I.

Paths are then trivially dependent functions from the interval:

Path A x y  :=  Pi I ({_} -> A)

refl   := {A, x, i} -> x
cong   := {A, B, f, p, i} -> f (p i)
sym    := {A, p, i} -> p (I_inv i)
funext := {A, B, h, i, a} -> h a i

Function extensionality — which requires an axiom in intensional MLTT — falls out as a one-line definition. The endpoints x, y in Path A x y are documentation; they are recovered by applying the path to I_zero and I_one.

Transport, composition, and Glue

Transport, transp P x : P\,I_one given x : P\,I_zero, is implemented by dispatch: each type-former supplies a transp_fn slot. Discrete types (Bool, Nat) use identity; pairs recurse; Pis use the standard contravariant/covariant scheme. This is, in the proposal's words, "the same shape Disp already uses for predicate_frame (recognizer dispatch with H-rule fallback) and eliminator_frame (case dispatcher with stuck-mint fallback). Cubical operations are not new in kind; they are new instances of an existing pattern." (CUBICAL_PROPOSAL.typ:168–173).

The Glue type-former is the trick that makes univalence compute. Glue B φ T e is "almost B, but with faces specified by φ replaced by T via the equivalence e." Its transp_fn recognizes ua-shaped paths and applies the stored equivalence directly:

ua : (A ≈ B) -> Path Type A B
ua := {A, B, e, i} -> Glue B (i I_or (I_inv i)) A e

So transp (ua not_equiv) TT evaluates to not_equiv.forward TT = not TT = FF. Univalence is a definable operation, not an axiom: equivalent types compute as equal.

Quote (CUBICAL_TYPE_THEORY_FOR_DUMMIES.typ:544–554) — "In MLTT, there was no way to give it computational content... Cubical solves this by providing Glue as a type former whose values encode equivalences directly, and whose transport rule unpacks them."

7.3  The long arc: synthesis and bootstrap

The final motivation, from GOALS.md:

"I want to synthesize the best possible programs given a specification... Turn type checker into a function that takes a program and returns 1 or 0, combine that with functions that measure performance or program size, and somehow turn all that into a function represented purely in some low-level deterministic turing-complete calculus." — GOALS.md:5–6

Once the kernel is small enough to live inside its own object language — and once enough mathematical structure (categorical, cubical) is library-defined within it — one has the prerequisites for neural-guided synthesis: an external optimizer can generate candidate trees, the type-checker (itself a tree) can score them, and the loop can refine itself. The path is:

  1. Tree calculus (done) gives a Turing-complete, reflective substrate.
  2. Seven-primitive kernel (done) gives dependent types as library code.
  3. Categorical typing (proposed) gives mathematical legibility of library structure.
  4. Cubical extensions (proposed) give univalence and HITs without axioms.
  5. Self-hosting checker (in progress) closes the metacircular loop.
  6. Optimizer-as-tree-program (future) enables self-improving synthesis.

The thesis: a small, reflective, parametric core, with all mathematical content lifted into library code, is the practical path to a self-improving theorem-proving language. Whether the long-term vision is realized or not, the structural commitments — types are predicates, the kernel is minimal, the discipline is metacircular — make Disp a worthwhile object of mathematical study on its own.

8.  Glossary

Tree calculus.
An abstract rewriting system on values built from one nullary constructor (the leaf △), with binary application. The reduction rules K, S, and triage make it Turing complete and structurally complete — every program can decide the shape of every value by reduction alone.
Triage.
The third reduction rule of tree calculus: a function of shape △(△cd)b dispatches on the shape of its argument (leaf, stem, or fork). Triage is the primitive reflection mechanism.
Hash-consing.
The implementation technique of storing each distinct tree exactly once and identifying trees by integer ID. Makes structural equality an O(1) identity check.
Bracket abstraction.
The algorithm [x].M that removes a free variable from an expression, leaving a closed combinator expression in I, K, S (plus partial applications). Bridges the surface λ-syntax to tree calculus.
Types-as-predicates.
Disp's design principle: a type is a tree-program T : Tree → Bool, and v : T iff T(v) = TT.
Scott encoding.
Encoding a data type as its own eliminator: a Boolean is a 3-argument function that selects one of its branches. Removes the need for primitive case opcodes.
Neutral / hypothesis.
A kernel-minted tree of shape wait(kernel.hyp_reduce)(meta) standing for "an unknown value of stored type T." Used to type-check under binders.
Parametric walker.
The user-mode reduction discipline. The five reduction rules with two restrictions: no triage on a hypothesis, and no stem-rule fabrication of a fork whose head matches a kernel-minted neutral signature.
Raw mode.
The five reduction rules without restrictions. Privileged: kernel handlers run in raw mode.
Action protocol.
Tagged return values Extend(B) / Return(v) from a type's codomain function. Let the kernel evaluate hypothesis applications without knowing the type's internal structure.
H-rule (hypothesis rule).
The rule inside predicate_frame: if the candidate is a hypothesis, compare reconstructed type against stored type rather than running the recognizer. Sound because hash-consing makes reconstruction-equal-stored an O(1) identity check.
Stuck neutral.
A neutral produced by eliminator_frame when an eliminator's target is a hypothesis. Stored type is the motive applied to the target; payload is the original target.
Escape check.
The check performed by bind_hyp on its body's result: if the minted hypothesis is reachable via any non-neutral path, the binder rejects. Prevents hypothesis leakage from closed proofs.
Guard scan.
The public-boundary check inside guard: rejects candidates containing pre-existing kernel-minted neutrals. Closes the "smuggle a hypothesis" soundness attack.
Path.
(Cubical, proposed.) A dependent function Pi I ({_} -> A) from the interval into a type. Endpoints recovered by applying the path to I_zero and I_one.
Glue.
(Cubical, proposed.) A type former whose values encode equivalences directly. Its transport rule applies the stored equivalence, giving univalence computational content.
Univalence.
The principle that equivalent types are equal. In Disp's cubical extension this becomes a theorem, not an axiom, because the ua path is constructed using Glue with computational transport behavior.
